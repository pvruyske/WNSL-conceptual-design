# PROMAR Project Life Cycle — Comprehensive Summary

## 1. Overview

The PROMAR Project Life Cycle defines the structured approach used to deliver projects at delaware.  
It consists of **five main stages** and an overarching **Monitor & Control** process.  
Each stage is subdivided into **phases**, separated by **quality gates** that ensure formal validation and closure before moving forward.

### Main Stages

1.  **Discover** (Pre‑project)
2.  **Initiate**
3.  **Explore**
4.  **Execute**
5.  **Deploy**
6.  **Run**

**Monitor & Control** spans the entire project.

## 2. Stages and Their Phases

### **Discover (Pre‑project)**

Purpose: Align customer and delaware on objectives, value, and feasibility.

**Phases**

*   **Pre‑project**:
    *   Pre‑sales collaboration
    *   Define business value, benefits, scope
    *   High‑level requirements, budget, timeline
    *   Proposal, rate table, SOW, MSA
    *   Initial stakeholder alignment

**Outcomes**

*   Clear business value and objectives
*   Agreed scope and requirements
*   Contractual baseline established

### **Initiate**

Purpose: Mobilize the project team, set governance, and prepare the Explore phase.

**Phases**

*   **Preparation**
*   **Initial Plan**
*   **Kick‑off / Startup**

**Key focus areas**

*   Project governance
*   Ways of working & acceptance criteria
*   Baseline planning
*   Roles, responsibilities, staffing
*   Risk register, change plan, communication plan
*   Workshop planning

**Outcomes**

*   Team mobilized
*   Governance model aligned
*   Explore phase ready to start

### **Explore**

Purpose: Gain process understanding, design the solution at high level, confirm scope.

**Phases**

*   **Inception**
    *   Customer Process Discovery (CPD)
    *   Conceptual Design (CCD)
*   **Fit‑to‑Standard / Blueprint (F2S / BBP)**

**Key focus areas**

*   Deep process understanding
*   Create BPML (Business Process Master List)
*   Conceptual decisions influencing blueprint scope
*   Fit/gap analysis
*   Blueprint documentation
*   Initial backlog (WBS) creation

**Outcomes**

*   Scope confirmed
*   High‑level design validated
*   Clear blueprint for Realization
*   Backlog for Realization prepared

### **Execute**

Purpose: Build the solution, test integrations, validate readiness for deployment.

**Phases**

*   **Realization (REA)**
*   **System Integration Testing (SIT)**
*   **Business Readiness Testing (BRT/UAT)**

**Realization**

*   Iterative sprint‑based build (3–6 weeks)
*   Detailed design sessions
*   Configuration, development, unit testing
*   Integration alignment
*   Sprint ceremonies (planning, stand‑up, review, retrospective)
*   Initial cutover planning
*   FIT and XFIT (Functional Integration Testing)

**SIT**

*   Integrated scenario testing
*   Multiple SIT cycles
*   Bug resolution & triage
*   Prepare training and documentation
*   Validate security and migrated data

**BRT / UAT**

*   End‑user execution of real business scenarios
*   Validate readiness for daily operations
*   Finalize cutover and deployment plans

**Outcomes**

*   Integrated and tested solution
*   Data migration rehearsed
*   Organization ready for training & change
*   Clear Go/No‑Go decision for Final Preparation

### **Deploy**

Purpose: Execute the cutover to production and support initial operations.

**Phases**

*   **Final Preparation**
*   **Go Live**
*   **Hypercare**

**Key focus areas**

*   Final data migration
*   Cutover execution
*   Golden orders monitoring
*   KPI monitoring
*   End‑user training completion
*   Rapid defect resolution

**Outcomes**

*   Solution deployed
*   Business is using the system
*   Hypercare concluded successfully

### **Run**

Purpose: Transfer to support organization and stabilize operations.

**Phases**

*   **Operate / Run & Accelerate (KIR)**
*   **Project Closing**

**Key focus areas**

*   Support handover
*   Lessons learned
*   Documentation harvesting
*   Final acceptance
*   Offboarding of project team

**Outcomes**

*   System in steady‑state operations
*   Support organization fully takes over
*   Project formally closed

## 3. Quality Gates

Quality Gates (QG) mark the transition between phases and stages.  
They ensure:

*   **Clarity and alignment** on expectations
*   **Progress tracking** via objective criteria
*   **Formal closure** of previous phase
*   **Billing milestones** for fixed‑price projects

### Examples of QG Criteria (selection)

*   CPD validated
*   Blueprint documents signed off
*   Backlog prepared
*   Sprint planning completed
*   Test plans and scripts ready
*   SIT pass rate > 90%, no critical bugs
*   Cutover plan validated and tested
*   End‑user training completed
*   Production system ready
*   Hypercare completed and handover done
*   Project closure document signed

## 4. Key Activities per Phase

### Preparation / Initial Plan / Kick‑off

*   Governance setup
*   Tools and procedures definition
*   OCM, data migration, and integration strategies
*   Scope and timeline clarification
*   Risk and communication plans

### CPD / CCD / F2S / BBP

*   CPD workshops
*   Blueprint workshops
*   Conceptual design decisions
*   Fit/gap identification
*   Creation of BPML
*   Blueprint documentation and sign‑off

### Realization

*   Detailed design workshops
*   Configuration
*   Custom development
*   Unit testing
*   Sprint ceremonies
*   Planning for cutover and testing

### SIT

*   SIT execution across domains
*   Bug triage and reporting
*   Test data and integration validation

### BRT / UAT

*   Testing by end‑users
*   Training material creation
*   UAT documentation and sign‑off

### Final Preparation + Go Live

*   Final migration and cutover
*   System readiness validation
*   Hypercare support

### Run

*   KPI monitoring
*   Handover to support
*   Lessons learned
*   Project acceptance and closure

## 5. Key Deliverables per Phase

### Preparation

*   Governance structure
*   Risk register
*   Data migration & integration strategies
*   GDPR requirements
*   Acceptance criteria
*   Deliverables list
*   Project standards

### Initial Plan

*   Planning with milestones
*   Workshop schedule
*   Blueprint backlog

### CPD / CCD / Blueprint

*   BPML
*   Conceptual design
*   Fit/gap list
*   Blueprint documents (signed off)
*   Backlog for Realization
*   Release calendar

### Realization

*   Config documentation
*   Specs for customizations
*   Built solution
*   Updated cutover plan
*   Sprint outputs
*   Go/No‑Go for SIT/BRT

### SIT / BRT

*   Executed test cases
*   SIT report and UAT report
*   Security and authorization validation
*   End‑user training material

### Final Preparation / Go Live

*   Final cutover plan
*   Migrated data
*   Golden orders
*   Go Live checklist completion

### Run

*   Lessons learned
*   Project closing document
*   Operational support handover

## 6. Summary of Stage Outcomes
